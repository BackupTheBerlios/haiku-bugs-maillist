 When system functions are made available as include files, they should
 also be made available as libraries. In other words, if a user does
 not need to add particular -I options on the gcc command line in
 order to get a header file, he also should not have to add a
 particular -L option in order to get access to the directory where
 the library is defined. And vice versa.

 <netdb.h> is in the default -I include path, but libnet.so is not in
 the default -L library path.

 I really don't want to add absolute directory names like
 /boot/beos/system/lib to gnulib's autoconf macros. The alternative
 is to document that configure should always be invoked with
 option LDFLAGS="-L/boot/beos/system/lib", but this is lame as well.

 I'm not sure whether it's gcc or ld which should be changed.
