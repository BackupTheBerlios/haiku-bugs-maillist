Comment (by phoudoin):

 > I think this problem is a bit implementation dependent, and GLTeapot
 bases its behavior on the
 > implementation: this would be a good reason to patch GLTeapot.
 > Thought?

 You're right, GLTeapot made the assumption that rendering is always done
 in a 32 bits RGB pixels buffer, so no color space conversion rounding
 errors could occurs.

 GLTeapot code is the faulty here, as such assumption in OpenGL should not
 be made.
 Notice, BTW, that BGLView don't offer a way to specify RGB colordepth.
 It's BGL_RGB or BGL_INDEX, period. On other platforms, there is often a
 way to ask for a 16, 24 or 32bits colorspace.
