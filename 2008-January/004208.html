Comment (by jackburton):

 Replying to [comment:3 stippi]:
 > Hi Stefano, thanks for picking this up. I investigated this one a bit
 the other day and it should have to do with Undo and the codepath for it.
 If you use TextView::Insert() directly, no Undo event is generated and
 this is the cause for the bug. I don't know if the original DeskCalc
 worked the same way as the current one, you probably need to make a test
 to find out if R5 TextView::Insert() generates an Undo step... if not, the
 problem is in DeskCalc, since BS just invokes Undo().

 Yes, thank you, I already studied a bit the code. AFAIK,
 BTextView::Insert() doesn't generate an Undo step, so I think we should
 just replace the BTextView::Undo() call with an appropriate
 BTextView::Delete() call.
